## 1. 项目概述与环境准备

### 1.1 项目来源

本项目 Fork 自 [Open Cowork](https://github.com/OpenCoworkAI/open-cowork) v3.3.1——一个开源的 AI Agent 桌面应用。Open Cowork 本身基于 Electron + React + TypeScript 构建，内置了对 Claude、DeepSeek、OpenAI、Gemini 等多种模型的支持。

我们在此基础之上，结合课程"Agent 高级开发"的要求，进行了深度二次开发，构建了三个核心模块：
1. **Coding Agent** — 具备读代码、改代码、执行命令、迭代反馈能力的 AI 编程代理
2. **Document Pipeline (RAG)** — 文档上传→解析→索引→向量检索→知识问答的完整文档知识库
3. **Skill 文档生成系统** — 基于 Skill 模板的自动化文档生成（PPT、Word、LaTeX、Python 分析代码等）

### 2.2 进程间通信（IPC）架构

项目使用 Electron 的 IPC 机制实现主进程和渲染进程之间的通信：

- **渲染进程 → 主进程（事件推送）**：通过 `contextBridge.exposeInMainWorld('electronAPI', ...)` 暴露的 `send()` 方法，发送 `client-event`
- **渲染进程 → 主进程（请求-响应）**：通过 `invoke()` 方法，调用 `client-invoke`，主进程处理后返回结果
- **主进程 → 渲染进程（服务器推送）**：通过 `server-event` 通道，推送 Agent 流式输出、Trace 步骤更新、会话变更等事件

### 2.3 数据流

```
用户输入消息
  → 渲染进程: ChatView.onSubmit()
  → Zustand: sessionStore.updateMessages()
  → IPC: electronAPI.invoke('client-invoke', { type: 'session.send', ... })
  → 主进程: MessageRouter.handleSessionSend()
  → SessionManager.processPrompt()
  → ClaudeAgentRunner.run()
  → pi-coding-agent: piSession.prompt()
  → pi-ai: streamOpenAICompletions()
  → DeepSeek API (HTTPS POST with streaming)
  ← SSE streaming response
  ← pi-ai: parseChunk() → thinking/text/tool_call deltas
  ← pi-coding-agent: subscribe() callback
  ← ClaudeAgentRunner: sendToRenderer()
  ← IPC: 'server-event' → 渲染进程
  ← Zustand: sessionStore.updatePartialMessage()
  ← ChatView: re-render with streaming content
```

3. `npm run rebuild` —— 为当前 Electron 版本重新编译 `better-sqlite3` 原生模块


### 3.3 配置 API 密钥

创建 `.env` 文件（或通过应用内的 Settings 面板配置）：

```env
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
OPENAI_BASE_URL=https://api.deepseek.com
OPENAI_MODEL=deepseek-v4-pro
```

配置存储在 `%APPDATA%/open-cowork/config.json` 中，支持多套配置切换（ConfigSet）。

### 3.4 启动开发模式

```bash
npm run dev
```

该命令执行的完整流程：
1. `unset ELECTRON_RUN_AS_NODE` —— 移除 VSCode 可能注入的环境变量（它会导致 Electron 启动失败）
2. `npm run download:node` —— 确保 Node.js 二进制已下载

5. `npm run build:mcp` —— 打包 MCP Server 到 `dist-mcp/`
6. `vite` —— 启动 Vite 开发服务器（含 vite-plugin-electron 构建主进程）

## 4. 构建步骤二：DeepSeek V4 模型集成

### 4.1 模型路由层 (`pi-model-resolution.ts`)

这一层负责将用户选择的模型名称（如 `deepseek-v4-pro`）解析为 pi-ai SDK 所需的 `Model` 对象。

**核心逻辑**：

```typescript
// 1. 构建候选 model ID 列表
function buildPiModelLookupCandidates(modelString: string) {
  // 尝试多种格式：原始 ID、自定义 provider/custom 前缀、openai/ 前缀等
  return [
    { provider: 'custom' as PiRegistryProvider, model: modelString },
    { provider: 'openai' as PiRegistryProvider, model: modelString },
    // ... 更多候选
  ];
}

// 2. 遍历候选，在 pi-ai 注册表中查找
export function resolvePiRegistryModel(modelString: string) {
  for (const candidate of buildPiModelLookupCandidates(modelString)) {
    const model = getModel(candidate.provider, candidate.model);
    if (model) return applyPiModelRuntimeOverrides(model, options);
  }
  // 未找到 → 构造 synthetic model
  return buildSyntheticModel(modelString);
}
```

**Synthetic Model 构造**：当模型不在 pi-ai 注册表中时（如自定义 API endpoint），动态构造 model 描述对象：

```typescript
function buildSyntheticModel(modelId: string) {
  return {
    id: modelId,
    provider: 'openai',
    api: 'openai-completions',
    baseUrl: config.baseUrl,  // 来自用户配置
    contextWindow: 128000,    // DeepSeek V4 上下文窗口
    input: ['text'],
    reasoning: false,
  };
}
```

### 4.2 Think Tag 解析器 (`think-tag-parser.ts`)

DeepSeek V4 在流式输出时通过 `reasoning_content` 字段返回思考过程。pi-ai SDK 将其转换为 `thinking_delta` 事件。但某些 API 端点（如 OpenRouter 或中转站）可能将思考内容内嵌在 `text` 中，使用 `<think>` / `</think>` XML 标签。

**解析策略**：
- 维护字符级状态机，检测 `<think>` 开始标签和 `</think>` 结束标签
- 标签包裹的内容标记为 `type: 'thinking'`
- 标签外的内容标记为 `type: 'text'`
- 未闭合的 `<think>` 块在流结束时作为 thinking 内容

```typescript
export function parseThinkTags(chunk: string): ThinkTagBlock[] {
  // 状态机: NORMAL → IN_THINK (遇到 <think>) → NORMAL (遇到 </think>)
  // 输出: { type: 'thinking', thinking: '...' } | { type: 'text', text: '...' }
}
```

### 4.3 Agent Runner 中的模型选择

在 `agent-runner.ts` 的 `run()` 方法中：

```typescript

  baseUrl: resolvedBaseUrl,

// → 返回 Model 对象 { id, provider, api, baseUrl, contextWindow, compat }
```

## 5. 构建步骤三：Coding Agent 核心引擎

### 5.1 概述

`ClaudeAgentRunner`（[src/main/claude/agent-runner.ts](ai-desktop/src/main/claude/agent-runner.ts)，约 3100 行）是整个 Coding Agent 的**核心执行引擎**。它负责：

- 接收用户消息，组装系统提示词
- 管理 Agent 会话生命周期（创建、复用、销毁）
- 桥接 MCP 工具到 pi-coding-agent SDK
- 处理流式响应（思考 → 文本 → 工具调用 → 工具结果）
- 实现权限控制（高风险操作需用户确认）
- 注入 Skills 和 Memory 到对话上下文

### 5.2 会话管理

Agent Runner 维护一个 `Map<string, CachedPiSession>` 缓存最多 25 个 SDK 会话：

```typescript
interface CachedPiSession {
  session: PiAgentSession;      // pi-coding-agent 会话实例
  thinkingLevel: string;         // 'off' | 'medium' | 'high'
  runtimeSignature: string;      // 运行时签名（用于判断会话是否可复用）
  ollamaNumCtx?: number;         // Ollama 上下文大小
}

private piSessions: Map<string, CachedPiSession> = new Map();
```

**会话复用逻辑**：
1. 检查是否有匹配 `sessionId` 的缓存会话
2. 计算 `sessionRuntimeSignature`（包含模型 ID、工作目录、工具定义等）
3. 如果签名匹配 → 复用会话（模型已缓存之前的对话上下文）
4. 如果签名不匹配 → 销毁旧会话，创建新会话

### 5.3 System Prompt 组装

系统提示词由多个组件拼接而成（约 4000+ tokens）：

```
1. 基础 Agent 指令（来自 pi-coding-agent 的默认系统提示词）
2. 角色定义（"你是一个 AI 桌面端助手..."）
3. 工具使用说明
4. Skills 列表（按需加载，只注入匹配当前任务的 Skill）
5. Memory 注入（从持久化 Memory 系统读取的相关记忆）
6. 工作目录信息
7. 冷启动时的对话历史（<conversation_history> XML preamble）
```

**冷启动历史注入**：当 SDK 会话被销毁后重新创建时，Agent Runner 从数据库中读取历史消息，序列化为 XML preamble：


### 5.4 流式响应处理

Agent Runner 通过 `piSession.subscribe()` 订阅 SDK 的流式事件：

```
subscribe((event) => {
  switch (event.type) {
    case 'turn_start':
    case 'turn_end':
    case 'message_start':
    case 'message_end':  → 构建 contentBlocks，保存消息到数据库
    case 'message_update': → 增量更新（thinking_delta / text_delta / toolcall_delta）
    case 'tool_execution_start': → 执行工具前检查权限
    case 'tool_execution_end':   → 收集工具结果
    case 'auto_compaction_start': → 上下文压缩
    case 'auto_compaction_end':
    case 'agent_end': → 清理、保存 session 摘要
  }
});
```

## 6. 构建步骤四：IPC 通信层

### 6.1 Preload 脚本

`src/preload/index.ts` 使用 `contextBridge.exposeInMainWorld` 暴露安全的 API 给渲染进程：

```typescript
contextBridge.exposeInMainWorld('electronAPI', {
  // 事件推送（单向）
  send: (event: ClientEvent) => ipcRenderer.send('client-event', event),

  // 请求-响应
  invoke: (event: ClientEvent) => ipcRenderer.invoke('client-invoke', event),

  // 服务器事件订阅（主进程 → 渲染进程）
  onServerEvent: (callback) => { /* 监听 'server-event' */ },

  // 各功能模块方法
  config: { get(), save(), test(), ... },
  mcp: { getServers(), getTools(), ... },
  skills: { getAll(), setEnabled(), ... },
  shell: { openExternal(), showItemInFolder() },
  // ...
});
```

**安全设计**：
- 不直接暴露 `ipcRenderer` 对象，而是封装为白名单方法
- 每个方法有类型声明，TypeScript 在编译期检查调用合法性
- URL 参数经过 `safeUrl` 校验，防止任意 URL 打开

### 6.2 主进程 IPC Handler

在 `src/main/index.ts` 中注册约 40 个 IPC handler：


## 7. 构建步骤五：Document Pipeline (RAG)

### 7.1 架构

Document Pipeline 是一个 6 阶段的文档知识库流水线：

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  Store   │→  │  Parser  │→  │ Indexer  │→  │ Vector   │→  │   QA     │→  │  Wiki    │
│ (存储)   │   │ (解析)   │   │ (索引)   │   │ Store    │   │ (问答)   │   │ Builder  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### 7.2 阶段详解

**阶段 1：DocumentStore** (`document-store.ts`)
- 管理文档的文件系统存储
- 维护文档元数据（名称、类型、大小、上传时间）
- 基于文件路径的 CRUD 操作

**阶段 2：DocumentParser** (`document-parser.ts`)
- 支持格式：PDF、DOCX、TXT、Markdown、XLSX
- PDF 解析采用**三层回退策略**：
  1. **pdfplumber**（首选）：表格提取能力强，中文支持好
  2. **PyMuPDF (fitz)**（备选）：速度快，文本提取质量高
  3. **pypdf**（兜底）：纯 Python 实现，无系统依赖
- 文本分块：按段落/标题分块，保持语义完整性

```python
# document-parser.ts 中调用的 Python 解析逻辑（简化）
def parse_pdf(file_path: str) -> str:
    # Layer 1: pdfplumber
    try:
        import pdfplumber
        with pdfplumber.open(file_path) as pdf:
            return '\n'.join(page.extract_text() for page in pdf.pages)
    except Exception:
        pass

    # Layer 2: PyMuPDF
    try:
        import fitz
        doc = fitz.open(file_path)
        return '\n'.join(page.get_text() for page in doc)
    except Exception:
        pass

    # Layer 3: pypdf (fallback)
    from pypdf import PdfReader
    reader = PdfReader(file_path)
    return '\n'.join(page.extract_text() for page in reader.pages)
```

**阶段 3：DocumentIndexer** (`document-indexer.ts`)
- 将解析后的文本块（Chunk）转为向量
- 调用 DeepSeek Embedding API 进行向量化
- 管理索引元数据（chunk_id、文档来源、位置偏移）

**阶段 4：VectorStore** (`vector-store.ts`)
- 基于内存的向量存储（使用 cosine similarity）
- 支持 CRUD 操作和相似度搜索
- 向量维度：DeepSeek Embedding 输出 4096 维

```typescript
class VectorStore {
  async search(query: string, topK: number = 5): Promise<SearchResult[]> {
    // 1. 将 query 转为向量
    const queryEmbedding = await this.embed(query);
    // 2. 计算与所有存储向量的 cosine similarity
    // 3. 返回 top-K 结果（含分数和原始文本块）
  }
}
```

**阶段 5：DocumentQA** (`document-qa.ts`)
- 接收用户问题，进行 RAG 问答
- 流程：query embedding → vector search → rerank → LLM 生成答案
- 答案包含证据引用（来源文档 + 相似度分数）

**阶段 6：LLMWikiBuilder** (`llm-wiki.ts`)
- 将文档知识库可视化为 Wiki 卡片
- 使用 LLM 生成文档摘要、关键概念、结构视图

### 7.3 Orchestrator

`DocumentPipeline` 类（`index.ts`）将所有阶段统一编排：

```typescript
class DocumentPipeline {
  readonly store: DocumentStore;
  readonly parser: DocumentParser;
  readonly indexer: DocumentIndexer;
  readonly qa: DocumentQA;
  readonly wiki: LLMWikiBuilder;
  readonly vectors: VectorStore;

  // 完整工作流
  async uploadAndIndex(filePath: string): Promise<DocumentMeta> {
    const meta = await this.store.add(filePath);
    const chunks = await this.parser.parse(filePath);
    await this.indexer.index(meta.id, chunks);
    return meta;
  }
}
```

## 12. 关键技术问题与解决方案

### 12.1 问题：thinking content block 导致 400 错误

**现象**：
Coding Agent 在第二轮对话时报错：
```
400 Bad Request: unknown variant 'thinking', expected 'text' at line 1 column 18577
```

**根因分析**：
- pi-ai SDK 将 DeepSeek V4 的思考过程存储在 `{ type: 'thinking', thinking: '...' }` content blocks 中
- 虽然 DeepSeek V4 在**响应**中返回 `reasoning_content`，但其**请求**的 messages content[] 数组只接受 `text` 类型
- `pi-model-resolution.ts` 中强制设置了 `requiresThinkingInContent: true`，导致 SDK 在发送历史消息时包含 `{ type: 'thinking', ... }` 块
- pi-ai SDK 的 `detectCompat()` 也对 DeepSeek V4 自动检测为 `requiresThinkingInContent: true`

**解决方案**（[pi-model-resolution.ts:331-344](ai-desktop/src/main/claude/pi-model-resolution.ts#L331)）：
```typescript
// 修改前：
if (nextModel.api === 'openai-completions' && DEEPSEEK_V4_MODEL_PATTERN.test(nextModel.id)) {
  compat: { requiresThinkingInContent: true }
}

// 修改后：
if (nextModel.api === 'openai-completions' && DEEPSEEK_V4_MODEL_PATTERN.test(nextModel.id)) {
  compat: {
    requiresThinkingInContent: false,  // 不在 content[] 中发送 thinking 块
    requiresThinkingAsText: true,       // 将 thinking 转为 { type: 'text', text: '...' }
  }
}
```

**效果**：SDK 在 `convertMessages()` 中将 thinking 块转为普通文本：
```javascript
// pi-ai SDK: openai-completions.js
else if (compat.requiresThinkingAsText) {
  const thinkingText = nonEmptyThinkingBlocks.map(b => b.thinking).join("\n\n");
  assistantMsg.content = [
    { type: "text", text: thinkingText },
    { type: "text", text: textContent },
  ];
}
```

### 12.2 问题：PDF 解析兼容性

**现象**：部分 PDF 文件解析失败或返回乱码。

**根因分析**：
- pdfplumber 依赖 pdfminer，某些 PDF 的 xref 表损坏导致解析失败
- PyMuPDF 需要安装系统级依赖（mupdf），在 Windows 上可能缺失
- pypdf 兼容性最好但文本提取质量可能不如前两者

**解决方案**（[document-parser.ts](ai-desktop/src/main/document-pipeline/document-parser.ts)）：
实现三层回退策略，确保至少有一种解析器能工作。详见第 7.2 节。

### 12.3 问题：Windows 下 `unset` 命令不支持

**现象**：`npm run dev` 在 Windows 终端失败：
```
'unset' is not recognized as an internal or external command
```

**根因分析**：`package.json` 的 dev script 使用了 Unix 的 `unset` 命令，在 Windows cmd/PowerShell 中不可用。

**临时解决方案**：
```bash
# 在 Windows 上手动设置环境变量为空，跳过 unset 步骤
ELECTRON_RUN_AS_NODE= && npm run download:node && npm run build:mcp && npx vite
```

### 12.4 问题：Session 会话复用中的签名变化

**现象**：切换工作目录后，Agent 丢失对话上下文。

**根因分析**：会话签名（`sessionRuntimeSignature`）包含工作目录路径。切换目录后签名变化，Agent Runner 判断会话不可复用，创建新会话。

**解决方案**（[agent-runner.ts:1890-1945](ai-desktop/src/main/claude/agent-runner.ts)）：
新会话启动时，通过 `serializeMessageContentForHistory()` 将历史消息注入为 `<conversation_history>` XML preamble，保留之前的对话上下文。

## 结语

本项目基于 Open Cowork v3.3.1，在其成熟的 Electron 桌面框架上，完成了三个核心模块的构建：

1. **Coding Agent**（3100 行 agent-runner.ts）：具备多轮对话、工具调用、权限控制、流式响应、Trace 跟踪的完整 AI 编程代理
2. **Document Pipeline RAG**（8 文件，3100 行编排器）：6 阶段文档知识库，支持多格式解析、向量检索、LLM 问答
3. **Skills 文档生成系统**（999 行管理器 + 14 个内置 Skill）：模板化自动文档生成，支持热重载

关键技术突破：
- 解决了 DeepSeek V4 API 的 thinking content block 兼容性问题
- 实现了 PDF 解析的三层回退策略
- 设计了基于会话签名的 Agent 上下文保持机制
- 构建了 40+ IPC 通道的进程间通信层

项目完全符合课程"Agent 高级开发"的三个核心要求，并在此基础上进行了深度工程化实践。
