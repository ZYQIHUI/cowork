5. 构建 Coding Agent（二）—— 实验步骤 2：Reasonix 缓存优化

目标：最大化 DeepSeek API 的 prompt cache 命中率，降低 token 消耗和延迟

参考 deepseek-reasonix 的设计理念。DeepSeek API 的缓存机制要求前缀完全匹配才能命中——这意味着每次请求的前缀内容（system prompt、工具定义、仓库摘要）必须保持稳定。

策略 1：Append-Only Session

会话消息只追加不修改。新消息追加在末尾，历史消息前缀不变。这样每次 API 调用时，缓存的前缀部分（历史消息+系统提示）可以直接命中。

实现：SessionManager 的消息数组采用不可变 push，禁止 splice/修改已有消息

策略 2：Stable Prefix Prompt

系统提示词、工具 schema 定义、仓库摘要信息固定在对话历史的最前面，顺序和内容都不变。

实现：buildPiSessionRuntimeSignature() 中固化工具定义顺序，AGENTS.md 内容放前缀

策略 3：Rolling Task State

长任务中，将工具调用结果压缩为滚动摘要，避免上下文无限膨胀。只保留最近 N 轮的完整工具结果，更早的压缩为摘要。

实现：error/log 压缩 + 上下文窗口监控 + 超限自动摘要

策略 4：Cache Hit 监控

实时统计每次 API 调用的 cache_hit_tokens / cache_miss_tokens，可视化展示节省的 token 数量和成本。

实现：CacheMonitorExtension 注入到 Agent Runtime，数据通过 IPC 实时推送到 UI

缓存命中原理

Request N:   [System Prompt] [Tool Schemas] [Repo Summary] [Turn 1] [Turn 2] ... [Turn N-1] [Turn N]
Request N+1: [System Prompt] [Tool Schemas] [Repo Summary] [Turn 1] [Turn 2] ... [Turn N-1] [Turn N] [Turn N+1]
                                    ↑ 前缀完全一致 → Cache HIT ↑                          ↑ 新内容 → Cache MISS ↑

6. 构建 Coding Agent（三）—— 实验步骤 3：CodeGraph 仓库理解

目标：让 Agent 理解仓库级结构，精准检索上下文，减少无效文件读取

参考 codegraph 的设计思路。没有 CodeGraph 时，Agent 只能逐个读取文件"摸索"代码结构，浪费大量 token。有了 CodeGraph，Agent 可以直接查询"这个函数被谁调用"、"修改这个文件会影响哪些测试"。

五层代码知识图谱

图层构建方式回答的问题
Symbol Graphtree-sitter / LSP 静态分析这个函数/类在哪里定义？有哪些引用？
Call Graph函数调用关系解析这个函数调用了谁？被谁调用？调用链是什么？
Import Graphimport/require 语句解析这个模块依赖哪些文件？哪些文件依赖它？
Test Graph测试文件 ↔ 源文件映射修改这个源文件需要跑哪些测试？
Build Graph构建配置解析这个目标依赖哪些源文件？编译顺序是什么？

CodeGraph 工作流程

  1. 预处理：项目启动时扫描仓库，构建五层图谱索引
  2. 存储：图谱数据存入 better-sqlite3，支持高效查询
  3. 查询接口：Agent 通过 MCP Tools 查询图谱（如 get_callers, get_imports）
  4. 上下文注入：查询结果自动注入到 Agent 的上下文窗口
  5. 错误定位：根据错误日志和测试失败，沿调用链自动定位候选文件

Token 节省效果：不用把整个仓库文件列表放进 prompt，只放相关的 3-5 个文件路径和依赖关系

与传统方式的对比

对比维度传统方式（无 CodeGraph）CodeGraph 方式
理解代码结构逐个 ls + read 文件摸索 → 多轮对话一次查询即获取依赖全景 → 1 轮对话
定位相关文件Agent 凭"直觉"猜测 → 可能漏掉关键文件沿 import/call graph 精准定位 → 不遗漏
修改后测试可能跑全量测试 → 浪费时间Test Graph 定位相关测试 → 精准测试
Token 消耗大量 ls/read 调用消耗 token单次查询返回结构化结果 → 节省 60%+

8. 构建文档流水线（二）—— 索引与向量存储

目标：将解析后的文档内容切块、向量化、建立可检索索引

DocumentIndexer：智能分块策略

  • 分块大小：maxChunkChars = 1000 字符，overlapChars = 100 字符重叠。重叠保证跨块边界的语义连续性
  • CJK 感知：estimateTokenCount() 区分中文字符 (×1.5) 和英文 (×0.3)，准确估计 token 数
  • 三类块：text（正文段落）、table（表格的 markdown 序列化）、image（OCR 文字 + 图注）
  • 批量嵌入：embeddingBatchSize = 8，并行调用嵌入 API，支持进度回调

VectorStore：混合检索引擎

自建内存向量存储（~520 行），零外部依赖。

  • 存储：Map<chunkId, Chunk> + JSON 文件持久化
  • 嵌入 API：调用 DeepSeek 兼容的 /v1/embeddings，模型 text-embedding-3-small (1536 维)
  • 向量检索：cosineSimilarity() 计算，纯 JS 实现
  • BM25 检索：computeBM25Score() (k1=1.5, b=0.75)，tokenize() 支持中英文混合分词
  • 混合搜索：Reciprocal Rank Fusion (RRF, k=60) 融合向量和关键词两路得分

混合检索的 RRF 融合算法

// 并行执行向量检索和 BM25 关键词检索，用 RRF 融合排名
const [vectorResults, keywordResults] = await Promise.all([
  this.search(queryEmbedding, candidateLimit),    // 余弦相似度
  this.keywordSearch(query, candidateLimit),       // BM25 评分
]);

// RRF: score = 1/(k + rank), k=60
for (let i = 0; i < vectorResults.length; i++)
  entry.rrfScore += 1 / (60 + i + 1);  // 向量排名贡献
for (let i = 0; i < keywordResults.length; i++)
  entry.rrfScore += 1 / (60 + i + 1);  // 关键词排名贡献

9. 构建文档流水线（三）—— RAG 问答与 LLM Wiki

目标：基于检索结果生成可信回答（含证据溯源），并将文档组织为结构化知识

DocumentQA：检索增强生成问答

工作流程：

  1. 用户提问 → searchHybrid() 检索 topK=10 个相关块
  2. 将检索到的 chunk 内容和证据信息组成 context prompt
  3. 发送给 DeepSeek V4 Pro 生成答案
  4. 返回 QAAnswer { answer, evidence[], confidence, tokensUsed }

两种模式：precise (temperature=0.1) 用于事实查找；conversational (temperature=0.5) 用于讨论性问答

证据溯源：EvidenceItem 包含 fileName + pageNumber + sectionTitle + tableId + cellRange + excerpt，每个回答都能追溯到原文具体位置

LLMWikiBuilder：结构化知识组织

与普通 RAG 的区别：RAG 每次问答都需要检索 → Wiki 预先生成结构化的知识卡片，可复用。

WikiCard 生成流程：

  1. 将文档完整内容发送给 LLM
  2. LLM 生成摘要、提取关键概念、生成章节摘要
  3. extractJsonBlock() 健壮提取 JSON 结果
  4. 缓存到 Map<docId, WikiCard>，后续直接复用
  5. findCrossReferences() 发现跨文档关联

目标：从 C 题 PDF 中提取问题背景、子问题、数据字段、约束条件等结构化信息

数学建模 C 题的 PDF 不是普通文档——它包含问题背景、多个子问题、附件数据说明、约束条件、评价指标、提交格式等固定结构的信息。需要设计专门的数据结构和 prompt 来提取。

CProblemParsed 数据结构设计

interface CProblemParsed {
  title: string;               // 题目标题
  background: string;          // 题目背景
  problems: CProblem[];        // 子问题列表
  attachments: CAttachment[];  // 附件信息
  dataFields: CDataField[];    // 数据字段描述
  constraints: string[];       // 约束条件
  evaluationMetrics: string[]; // 评价指标
  submissionFormat: string;    // 提交格式
  paperRequirements: string;   // 论文要求
}

interface CDataField {
  name: string;
  type: 'numeric'|'categorical'|'datetime'|'text';
  unit?: string;
  missingRate: number;  // 自动统计
  description: string;
}

DeepSeek 兼容性处理

问题：DeepSeek API 不支持 OpenAI 的 response_format: { type: 'json_object' } 参数

解决：

  • 在 system prompt 中明确要求"直接返回 JSON，不要用 ```json 包裹"
  • temperature 设为 0.1 提高输出稳定性
  • max_tokens 设为 4096 确保完整输出
  • 后处理：strip markdown 围栏 → JSON.parse → try-catch
  • 解析后自动补充字段（如 missingRate 默认值）

附件数据理解

对 C 题附件（Excel/CSV）进行自动字段分析：

  • 字段类型识别：numeric（数值型）/ categorical（分类型）/ datetime（时间型）/ text（文本型）
  • 缺失值统计：每列的 missingRate（缺失比例），为后续数据清洗提供依据
  • 统计特征：数值列计算 min/max/mean/std，分类型计算 uniqueValues
  • 数据字典生成：为每个字段生成结构化描述，供代码生成和论文写作使用

15. IPC 通信架构 —— 主进程与渲染进程的桥梁

目标：安全、类型安全地实现前后端通信

安全设计

  • contextIsolation: true — 渲染进程无法直接访问 Node.js
  • sandbox: true — 操作系统级沙箱隔离
  • nodeIntegration: false — 渲染进程不能 require Node 模块
  • 白名单校验：ALLOWED_CLIENT_EVENTS Set 包含 30+ 允许的通道
  • contextBridge：仅暴露 window.electronAPI，不暴露 ipcRenderer

通信模式

Renderer → Main：

  • window.electronAPI.invoke(event) — 请求-响应模式（带返回值）
  • window.electronAPI.send(event) — 单向推送（无返回值）

Main → Renderer：

  • mainWindow.webContents.send('server-event', event) — 推送实时状态
  • 流式消息、工具调用状态、缓存统计、解析进度等实时推送

18. 关键技术难点与解决方案

开发过程中遇到的主要技术挑战

#难点原因解决方案
1 PDF 解析返回空内容 pdfplumber/PyMuPDF 未安装，原项目只支持这两种引擎 设计三级回退链：pdfplumber → PyMuPDF → pypdf。pypdf 为纯 Python 包 (6.12.2 已预装)，零外部依赖，确保在任何环境下都能提取文本
2 DeepSeek 不支持 JSON 结构化输出 DeepSeek API 不兼容 OpenAI 的 response_format: json_object 参数 改用 prompt 工程：在 system prompt 中明确要求 JSON 格式 + 降低 temperature 到 0.1 + 后处理剥离 markdown 围栏 + 健壮的 JSON.parse
3 ELECTRON_RUN_AS_NODE 环境变量导致 Electron 崩溃 VSCode 扩展设置的 env var 泄露到 Electron 子进程 在 vite.config.ts 的 onstart handler 中 delete process.env.ELECTRON_RUN_AS_NODE，同时在 package.json dev 脚本中 unset 该变量
4 Python 脚本的 shell 转义问题 inline Python 脚本通过命令行传递时，引号和特殊字符难以正确转义 改用 runPythonScriptFile()：将脚本写入临时 .py 文件再执行，彻底避免 shell 转义
5 中英文混合的 token 估算 简单按字符数估算 token 在中英混合文本中误差很大 实现 CJK 感知的 estimateTokenCount()：中文字符 ×1.5 + 非中文 ×0.3
6 Skill 系统 EvalPanel 初始显示 null useState 声明了变量但没有 setter 函数 补充 setSkillEval/setCProbEval setter，更新 runEval 解构三种结果
7 LLM 生成的 JSON 格式不稳定 模型有时会包裹 markdown 代码块，有时会多输出说明文字 实现 extractJsonBlock()：先尝试匹配 ```json 围栏 → 匹配 { } 最外层 → 回退原文

关键经验总结

  • 回退链设计：外部依赖不可靠时，必须设计多级回退，确保系统在最简环境下也能工作
  • Prompt 工程 > 参数依赖：不同模型的 API 参数兼容性不同，良好的 prompt 设计比依赖特定参数更可靠
  • 环境变量污染：桌面应用需要考虑开发工具（VSCode）对运行环境的影响
  • 数据结构先行：CProblemParsed / QAAnswer 等数据结构在设计 pipeline 之前定义，确保上下游接口一致
